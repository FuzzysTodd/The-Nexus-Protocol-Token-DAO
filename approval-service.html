Verification Providers

  • Tenderly — planned integration for bytecode verification, simulations, and state diffs.
  • Dune — planned integration for contract activity, anomaly detection, and treasury behavior.

Configure these in a central config file, e.g.: 

"verification": {
  "tenderly": {
    "enabled": true,
    "account": "YOUR_TENDERLY_ACCOUNT",
    "project": "YOUR_PROJECT",
    "apiKey": "YOUR_TENDERLY_API_KEY"
  },
  "dune": {
    "enabled": true,
    "apiKey": "YOUR_DUNE_API_KEY",
    "queries": {
      "treasurySignals": 1923353
    }
  }
}

The approval service should treat Tenderly + Dune as signal providers, not executors. They inform the operator; they never move funds.

  // Button event delegation for Approve / Reject / Details
  document.addEventListener('click', function (e) {
    const btn = e.target;
    if (!btn.dataset.id) return;

    const id = btn.dataset.id;
    const action =
      btn.classList.contains('btn-approve') ? 'APPROVE' :
      btn.classList.contains('btn-reject')  ? 'REJECT'  :
      btn.classList.contains('btn-details') ? 'DETAILS' :
      null;

    if (!action) return;

    // Console log output
    console.log(`NEXUS APPROVAL ACTION →`, {
      id,
      action,
      timestamp: new Date().toISOString()
    });

    // Optional: visual feedback
    if (action === 'APPROVE') {
      btn.textContent = 'Approved';
      btn.style.opacity = '0.7';
    }
    if (action === 'REJECT') {
      btn.textContent = 'Rejected';
      btn.style.opacity = '0.7';
    }
    if (action === 'DETAILS') {
      alert(`Details for: ${id}\n\nCheck console for full payload.`);
    }
  });
